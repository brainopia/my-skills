# Сравнение 5 планов реализации adaptive-planning / smart-plan skill

## Что я сравнил

1. `omp-5.4/PLAN.md`
2. `omp-opus/PLAN.md`
3. `pi-5.4-pro-high/execplan/adaptive-planning-skill.md`
4. `pi-execplan-5.4/execplan/adaptive-planning-skill.md`
5. `pi-execplan-opus/execplan/adaptive-planning-skill.md`

Дополнительно я сверил ключевые предпосылки с реальными материалами по Pi:
- `docs/skills.md`
- `extending-pi/skill-creator/SKILL.md`
- `extending-pi/skill-creator/scripts/validate_skill.py`
- `execplan/references/PLANS.md`

Это важно, потому что часть планов написана под **OMP**, а часть — под **Pi**, и без проверки документации их легко сравнить некорректно.

---

## Главное наблюдение перед сравнением

Эти 5 файлов формально про один и тот же замысел, но на самом деле делятся на **две разные семьи**:

- **OMP-планы**: `omp-5.4`, `omp-opus`
- **Pi-планы**: `pi-5.4-pro-high`, `pi-execplan-5.4`, `pi-execplan-opus`

Поэтому сравнение надо делать в двух слоях:

1. **Концептуальная сила** — насколько хорошо план понимает саму задачу адаптивного планирования.
2. **Реальная пригодность к исполнению именно в Pi** — насколько из этого документа можно почти сразу идти делать рабочий skill.

Если конечная цель — **реально построить Pi skill**, то OMP-планы полезны как источник идей, но **не как основной blueprint**.

## Как я оценивал планы

Я смотрел на 10 критериев:

1. Соответствие целевой платформе
2. Полнота концепции (режимы, blind spots, research, HITL, decomposition)
3. Конкретность workflow
4. Качество контракта live plan
5. Качество выбора режима
6. Реалистичность HITL
7. Реалистичность subagent/research модели
8. Качество validation / acceptance
9. Работа с контекстом, handoff и restart
10. Поддерживаемость skill через progressive disclosure

## Краткий вывод

**Лучший план как база для реальной реализации в Pi — `pi-execplan-opus`.** Он самый зрелый, самый конкретный и лучше всех переводит идею в практический workflow: там есть полноценный формат live plan, lifecycle статусов, хороший tier/rubric, использование `interview` для структурированного HITL, нормальная стратегия subagent-ов и внятные behavioral acceptance criteria. Из `omp-opus` стоит позаимствовать только уровень конкретности и идею quick reference table.

- **Основа:** `pi-execplan-opus`
- **Лучшие дополнительные идеи:** `pi-5.4-pro-high`
- **Лучший OMP-план как донор конкретики:** `omp-opus`

## Детальный разбор каждого плана

## 1) `omp-5.4/PLAN.md`

### Что в нём особенно сильного

Это самый "архитектурный" из всех файлов. Он не торопится к частностям и сначала очень правильно решает верхний уровень задачи:

- предлагает **одну общую форму плана** для всех нетривиальных режимов;
- отделяет **политику выбора глубины** от формата документа;
- хорошо формулирует **review gates**:
  - completeness gate
  - blind-spot gate
  - execution gate
  - escalation gate
- даёт зрелую рубрику выбора режима по нескольким осям:
  - ambiguity
  - novelty
  - blast radius
  - reversibility
  - research need
  - user input need
  - size
  - dependency complexity
- отдельно осмысляет trade-offs между:
  - pure markdown
  - markdown + structured config
  - code-centric runtime

Это делает документ ценным как **design rationale**. Он хорошо отвечает на вопрос: *какой должна быть общая философия skill family?*

### Где он слаб

Как только нужен переход от философии к реализации, документ резко слабеет:

- нет привязки к Pi как платформе;
- нет конкретного способа делать HITL;
- нет tool-specific recipes;
- helper roles (`researcher`, `critic`, `decomposer`, `supervisor/council`) существуют почти как абстрактные персонажи, а не как реальные workflow;
- нет worked examples;
- нет понятной mechanics context reset / restart-from-plan;
- live plan описан по разделам, но не доведён до **полноценного шаблона с lifecycle**.

### Итог по плану

Это не плохой план. Наоборот: это **сильная теоретическая рамка**. Но именно как рабочая основа он слишком абстрактен. Его нельзя брать как главный implementation plan, зато из него нужно обязательно забрать:

- review gates taxonomy;
- finding classification (`blocker` / `nice-to-have` / `watchout`);
- идею единого контракта плана + depth policy.

## 2) `omp-opus/PLAN.md`

### Что в нём особенно сильного

Это самый "боевой" документ. Если бы целевая среда была OMP, его можно было бы брать почти напрямую.

Сильные стороны:

- очень конкретные tier protocols;
- готовые рецепты с `oracle`, `librarian`, `reviewer`;
- ясная quick-reference таблица по режимам;
- хорошо читаемый Execplan format;
- правило **"when in doubt, pick one tier higher"** — простое, но практически полезное;
- хороший уровень операционности: видно, в каком порядке агент должен действовать.

Из всех 5 файлов именно здесь лучше всего чувствуется **ритм реальной работы**, а не только идея.

### Где он слаб

Проблема не в качестве мышления, а в platform mismatch:

- он построен на OMP-механике (`exit_plan_mode`, `local://PLAN.md`, `task { agent: ... }`);
- single-file подход противоречит лучшей Pi-практике progressive disclosure;
- live-plan format слишком тонкий по сравнению с ExecPlan-ориентированными Pi-вариантами;
- нет хорошего restart/handoff протокола;
- нет RPD;
- нет формализованного backpressure;
- mode selection проще и мельче, чем в лучших Pi-планах.

### Итог по плану

Очень полезный донор **прикладной конкретики**, но плохая основа для Pi. Если вы хотите реальную Pi-реализацию, то из него стоит взять:

- уровень конкретности рецептов;
- quick reference table;
- принцип tier-up при сомнении;
- паттерн параллельного быстрого обзора кодовой базы перед планированием.

Но архитектурно копировать его нельзя.

## 3) `pi-5.4-pro-high/execplan/adaptive-planning-skill.md`

### Что в нём особенно сильного

Этот план дал несколько лучших идей во всей выборке.

#### 1. RPD
Он один из немногих, кто правильно чувствует важность **быстрого problem framing** ещё до полноценного плана.

Поля RPD там удачные:
- Goal
- Constraints
- Repo Facts
- Unknowns
- Proof
- Chosen Mode

Это очень полезно, потому что заставляет не "сразу планировать", а сначала собирать минимальный когнитивный каркас задачи.

#### 2. Backpressure
Этот план лучше всех проговаривает, что planning system должна не только помогать, но и **тормозить агента в правильных местах**.

Особенно сильна логика по режимам:
- для P0 — проверить side effects / validation / rollback;
- для P1 — запретить код до plan + review + approval;
- для P2/P3 — нельзя двигаться дальше, пока неизвестности не закрыты исследованием или вопросами;
- для P4 — нельзя начинать child work без approved master plan.

Это очень важная идея. В большинстве планов HITL есть, но backpressure как система почти не оформлен.

#### 3. Restart-from-plan
Очень зрелая мысль: продолжение от планового файла должно быть **нормальным штатным режимом**, а не аварийной мерой.

#### 4. Meta overlay
Это, возможно, самая интересная концепция во всей подборке: если главная неопределённость в *способе решения задачи*, а не в *шаге реализации*, сначала нужен meta-plan, а уже потом implementation plan.

Для сложных и новых доменов это реально полезно.

### Где он слаб

Главный недостаток — план слишком часто остаётся на уровне **отличного замысла**, но не доходит до плотной спецификации.

Слабые места:

- много сильных концептов, но мало operational detail;
- почти всё важное вынесено в будущие reference-файлы, которые ещё не написаны;
- нет достаточно конкретного live-plan contract;
- нет конкретной mechanics для `interview`;
- нет конкретных шаблонов вызова `subagent`;
- много текста, который повторяет сам себя;
- есть риск перегрузить v1 слишком богатой мета-логикой.

### Итог по плану

Это очень сильный **стратегический план второго слоя**. Сам по себе — не лучший blueprint для реализации, но он содержит идеи, которые нельзя потерять:

- RPD
- backpressure
- restart-from-plan
- naming convention для plan files
- meta overlay как опциональная возможность v2
- fallback, если `subagent` недоступен

Если взять лучший практический план и обогатить его именно этим содержанием, получится очень сильный итоговый вариант.

## 4) `pi-execplan-5.4/execplan/adaptive-planning-skill.md`

### Что в нём особенно сильного

Это хороший, аккуратный, Pi-native каркас. Он уже думает в правильных категориях:

- frontmatter / description / activation;
- validator;
- `README.md`;
- references + templates;
- `pi --no-skills --skill`;
- behavioral scenarios;
- context reset через file-based handoff.

В нём правильная общая структура skill-пакета и понятная логика "ядро + справочные материалы + шаблоны".

### Где он слаб

Проблема в том, что он ощущается как **предыдущая менее развитая версия** `pi-execplan-opus`.

Чего ему не хватает:

- конкретного live-plan format;
- достаточно жёсткого mode-selection rubric;
- anti-patterns;
- полноценного self-review contract;
- хорошо развёрнутых worked examples;
- более убедительной детализации subagent strategies;
- RPD и явного backpressure.

### Итог по плану

Как черновой implementation skeleton он хороший. Но если выбирать один документ для реальной работы, то почти всё ценное из него уже есть в `pi-execplan-opus` — только лучше.

Поэтому этот файл полезен скорее как промежуточная стадия эволюции идеи, а не как конечная база.

## 5) `pi-execplan-opus/execplan/adaptive-planning-skill.md`

### Что в нём особенно сильного

Это самый зрелый и самый практический план во всей выборке.

#### 1. Лучший live-plan contract
Он единственный действительно доводит идею live plan до рабочего контракта:

- `Status`
- `Tier`
- `Created`
- `Last updated`
- `Purpose`
- `RPD`
- `Assumptions & Constraints`
- `Blind Spots Addressed`
- `Approach`
- `Validation & Acceptance`
- `Action Log`
- `Decision Log`
- `Surprises & Discoveries`
- `Self-Review`
- `Next Steps`
- `Handoff Note`

И ещё важнее — он описывает **lifecycle**:
- planning
- awaiting-review
- approved
- in-progress
- blocked
- done

Это переводит "живой план" из красивой идеи в управляемый объект.

#### 2. Правильный HITL для Pi
План явно использует `interview` как структурированный способ:
- показать план пользователю;
- собрать решения;
- задать пакет вопросов;
- не сваливать всё в хаотичный чат.

Это очень сильное решение. Именно здесь видно понимание реального Pi-инструментария.

#### 3. Хорошая субагентная модель
Вместо абстрактного "можно использовать второго агента" план описывает три понятных паттерна:
- single reviewer pass
- parallel council
- oracle query

Плюс есть рабочий handoff/context reset протокол.

#### 4. Сильная рубрика выбора режима
Там есть:
- 6 измерений оценки;
- escalation / de-escalation logic;
- anti-patterns;
- числовые question budgets.

Это резко повышает воспроизводимость поведения.

#### 5. Лучшие examples и acceptance criteria
План не ограничивается словами "проверить поведение". Он даёт реальные сценарии на каждый tier и говорит, что именно должно наблюдаться.

### Где он слаб

Слабости есть, но они уже не системные:

- документ длинный и местами повторяется;
- нет meta overlay как отдельной опциональной надстройки;
- backpressure проговорён слабее, чем в `pi-5.4-pro-high`;
- naming convention для plan files менее продумана;
- можно усилить политику дешёвого/дорогого использования council/subagents.

### Итог по плану

Это лучший файл в наборе как **реальная отправная точка**. Он уже достаточно конкретен, чтобы по нему писать skill, reference docs и шаблоны.

## Что особенно важно в сравнении всех пяти

## 1. Главная развилка — не качество текста, а platform fit

Если цель — Pi skill, то OMP-планы нельзя честно ставить на первое место, даже если местами они очень хорошие. Они говорят на другом языке runtime.

### Почему это важно

Pi в реальности опирается на:
- `SKILL.md` + frontmatter `name` / `description`
- progressive disclosure
- `references/` и `assets/`
- `subagent`
- `interview`
- локальную валидацию skill

А OMP-планы строятся вокруг:
- `exit_plan_mode`
- `task { agent: ... }`
- `local://PLAN.md`
- OMP-специфических ролей и механик

То есть **концептуально это близкие проекты, но operationally это разные миры**.

---

## 2. Лучшие планы отличаются не только полнотой, но и правильным уровнем детализации

Есть два плохих крайних случая:

- слишком абстрактно (`omp-5.4`)
- слишком "только намерения, детали потом" (`pi-execplan-5.4`, местами `pi-5.4-pro-high`)

Лучший результат там, где есть:

- ясная архитектура;
- понятный contract;
- готовая файловая структура;
- concrete behavior examples;
- но без ухода в extension-level сложность.

Этому лучше всего соответствует `pi-execplan-opus`.

---

## 3. Самая недооценённая идея во всей подборке — backpressure

Многие планы говорят про review, approval, blind spots. Но именно `pi-5.4-pro-high` лучше всех объясняет, что planning system должна **не только помогать формулировать путь, но и создавать нужное трение**.

Если это не встроить, skill легко превратится либо в:
- красивый генератор markdown-файлов,
- либо в формальность, которую агент обходит.

Поэтому я считаю, что **backpressure нужно обязательно интегрировать в итоговый гибрид**, даже если в базовом документе оно описано слабее.

---

## 4. Самый практичный UX-вывод — HITL должен быть структурированным

Лучший шаг `pi-execplan-opus` — использовать `interview` для:
- approval,
- изменения направления,
- сбора нескольких связанных решений,
- возможно, tier override.

Это намного лучше, чем "вывалить план в чат и ждать, пока пользователь что-то напишет".

## Моя рекомендуемая стратегия: не выбирать один план, а собрать лучший гибрид

## Основа

Взять **`pi-execplan-opus`** как главный implementation plan.

Почему именно его:
- Pi-native
- лучший live-plan contract
- лучший HITL через `interview`
- лучшая детализация subagent patterns
- хорошие worked examples
- лучший уровень готовности к реальной сборке skill

## Что обязательно добавить из `pi-5.4-pro-high`

### 1. RPD
Встроить короткий RPD в начало каждого режима как обязательный micro-step.

### 2. Backpressure per tier
Прописать не только что делать, но и **что запрещено делать раньше времени**.

### 3. Restart-from-plan как first-class path
Handoff не должен быть аварией. Он должен быть нормальной частью дизайна.

### 4. Naming convention для plan files
Например:
- `execplan/2026-03-08-guided-cli-verbose-flag.md`
- `execplan/payment-redesign/master-plan.md`
- `execplan/payment-redesign/m1-auth-flow.md`

### 5. Meta overlay как опциональную v2-возможность
Не делать обязательным в MVP, но явно оставить как расширение для сложных доменов.

## Что взять из `omp-5.4`

### 1. Review gates taxonomy
Очень сильное усиление качества self-review:
- completeness
- blind-spot
- execution
- escalation

### 2. Classification of review findings
Полезно разделять:
- blocker before approval
- nice-to-have
- execution-time watchout

Это даёт плану больше пользы, чем просто "review passed / failed".

## Что взять из `omp-opus`

### 1. Quick reference table
Очень полезно иметь одну компактную таблицу:
- режим
- нужен ли plan file
- нужен ли HITL
- можно ли задавать вопросы
- нужен ли research
- нужны ли subagents

### 2. Tier-up rule
Если сомневаешься между двумя режимами — временно выбирай более осторожный.

### 3. Паттерн параллельных быстрых разведывательных агентов
Для guided/research/feature режимов полезно запускать лёгкий parallel scouting по разным частям репозитория.

## Что, на мой взгляд, недодумано почти во всех пяти планах

Ниже — мои собственные предложения, которых в наборе не хватает или которые раскрыты недостаточно.

## 1. Tier override пользователем

Почти все планы говорят, что агент сам выбирает режим. Это хорошо, но мало.

Я бы добавил явный механизм:
- агент выбирает tier;
- кратко показывает, **почему именно его**;
- пользователь может:
  - подтвердить,
  - поднять tier,
  - опустить tier,
  - разрешить/запретить вопросы,
  - разрешить/запретить research.

Это особенно хорошо ложится на `interview`.

## 2. Policy for blocker remediation

Если self-review или subagent нашли blocker, что дальше?

Нужна явная политика:
- blocker → план не уходит на approval, пока не исправлен или явно не вынесен как risk;
- watchout → план можно вынести на approval, но отметить отдельным блоком;
- nice-to-have → не тормозит workflow.

## 3. Cost / latency awareness для council и subagents

В хорошем skill важно не только качество, но и цена по времени/контексту.

Я бы добавил простую политику:
- по умолчанию — без council;
- reviewer/oracle — только для high-uncertainty или high-blast-radius задач;
- council — только для реально спорных архитектурных развилок;
- если проблема решается одним focused scout + self-review, не раздувать процесс.

## 4. Fallback, если `interview` недоступен или не нужен

Лучший UX — structured form. Но нужен backup path:
- если `interview` недоступен, показать краткое summary + 2–4 нумерованных варианта в обычном сообщении;
- если guided-mode запрещает planning-time questions, approval still should be explicit.

## 5. Единый минимальный contract для P0

P0/direct/quickstrike часто описан как "сразу делай". Я бы это ужесточил.

Даже в P0 должен быть обязательный micro-contract:
- 1 предложение RPD
- 2–4 blind spots
- 1 validation statement
- 1 короткий self-review after execution

Иначе P0 быстро деградирует в импульсивное действие.

## 6. Naming режима `quickstrike`

Это не критично, но я бы подумал о более нейтральном названии:
- `direct`
- `fast-path`
- `instant`

`quickstrike` запоминается, но звучит чуть более "маркетингово", чем остальные режимы. Если важна серьёзная инженерная тональность, можно переименовать. Если важна меметичность и легко различимые имена — можно оставить.

## Какой выбор я бы рекомендовал в зависимости от цели

## Если цель — сделать лучший практический Pi skill v1

Берите:
- **базу из `pi-execplan-opus`**
- **RPD + backpressure + restart-from-plan из `pi-5.4-pro-high`**
- **review gates из `omp-5.4`**
- **quick-reference table и tier-up rule из `omp-opus`**

Это мой главный совет.

## Если цель — сделать самый быстрый MVP

Тогда база может быть проще:
- `pi-execplan-5.4` как skeleton
- добавить только:
  - нормальный live-plan contract,
  - `interview` для approval,
  - минимальный RPD,
  - 5 worked examples.

Но это будет weaker version по сравнению с вариантом выше.

## Если цель — сделать самую умную долгосрочную систему

Тогда:
- всё из рекомендованного гибрида,
- плюс `meta overlay` как v2,
- плюс более формальный cost policy для subagents,
- плюс, возможно, extension-level enforcement в будущем.

## Окончательный вердикт

Если мне нужно выбрать **один** файл, который сегодня лучше всего вести в реализацию, я выбираю:

# `pi-execplan-opus/execplan/adaptive-planning-skill.md`

Но если мне нужно выбрать **наилучший подход**, а не просто лучший одиночный документ, я выбираю такой путь:

1. **Взять `pi-execplan-opus` как базу**
2. **Влить в него RPD, backpressure, restart-from-plan и naming ideas из `pi-5.4-pro-high`**
3. **Влить review gates из `omp-5.4`**
4. **Влить quick reference table, tier-up rule и дух конкретности из `omp-opus`**
5. **Не копировать ничего OMP-specific и не переусложнять v1 structured config-слоем**

Именно этот гибрид, на мой взгляд, даёт лучший баланс между:
- практической реализуемостью,
- качеством мышления,
- контролем рисков,
- хорошим UX для пользователя,
- и устойчивостью long-running задач.

## Финальная рекомендация в одном предложении

**Для реальной Pi-реализации выбирай `pi-execplan-opus` как основу, но обязательно усили его идеями `pi-5.4-pro-high` (RPD, backpressure, restart-from-plan) и точечной строгостью из `omp-5.4` / `omp-opus`; это будет заметно сильнее любого из пяти планов по отдельности.**
